PostgreSQL Running Slow? A Step-by-Step Diagnostic Guide

Your PostgreSQL database is slow and you don't know why. This guide walks you through a systematic diagnosis process — from active query monitoring to EXPLAIN analysis, index auditing, and autovacuum configuration — so you can find and fix the real problem.

Your application is slow. Users are complaining. You've checked the app servers, the network, the CDN — everything seems fine. The database is the suspect, but where do you start?

PostgreSQL performance problems almost always come from a handful of root causes: missing or wrong indexes, table and index bloat, misconfigured autovacuum, connection exhaustion, or poorly-written queries hitting the wrong execution plan. This guide walks you through a systematic diagnostic process to find which of these is causing your slowdown.

Step 1: Check What's Running Right Now

Before digging into historical data, look at what's happening in your database at this moment. The pg_stat_activity view shows you every active connection and what it's doing.

-- Find long-running queries (over 5 seconds)
SELECT
  pid,
  now() - pg_stat_activity.query_start AS duration,
  query,
  state,
  wait_event_type,
  wait_event
FROM pg_stat_activity
WHERE (now() - pg_stat_activity.query_start) > interval '5 seconds'
  AND state != 'idle'
ORDER BY duration DESC;

Pay attention to the wait_event_type and wait_event columns. If you see queries waiting on Lock events, you have a lock contention problem. If you see IO waits, you're hitting disk. If the query itself is running (no wait event), it's a CPU-bound computation problem — usually a sequential scan on a large table.

Quick check: Run SELECT count(*) FROM pg_stat_activity WHERE state = 'idle in transaction'; — if this number is high (more than a handful), you likely have long-running transactions holding locks open. This is a common cause of cascading slowness.

Step 2: Find Your Slowest Queries with pg_stat_statements

If your database has pg_stat_statements enabled (it should be — add it to shared_preload_libraries if not), you can see aggregate statistics across all query executions since the last stats reset.

-- Top 20 slowest queries by total execution time
SELECT
  left(query, 100) AS query_snippet,
  calls,
  round((total_exec_time / calls)::numeric, 2) AS avg_ms,
  round(total_exec_time::numeric, 2) AS total_ms,
  round((100 * total_exec_time / sum(total_exec_time) OVER ())::numeric, 2) AS pct_total
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 20;

This tells you which queries are consuming the most database time overall. Focus on the queries with the highest total_ms first — these have the biggest impact on overall performance, even if individual executions are fast but frequent.

Step 3: Read the EXPLAIN Output

Once you've identified a slow query, run EXPLAIN (ANALYZE, BUFFERS) to see the actual execution plan and understand what PostgreSQL is doing.

EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT *
FROM orders o
JOIN customers c ON c.id = o.customer_id
WHERE o.status = 'pending'
  AND o.created_at > now() - interval '7 days';

What to look for in EXPLAIN output

  • Seq Scan on large tables: If you see Seq Scan on a table with millions of rows, you're missing an index. This is the most common cause of slow queries.
  • High "rows removed by filter": If PostgreSQL is fetching many rows but filtering most of them out, your index isn't selective enough or you need a partial index.
  • Hash Join with large memory spill: Look for Batches: N (where N > 1) — this means the hash join spilled to disk. You can increase work_mem for this session to fix it.
  • Nested Loop with many iterations: A nested loop over a large dataset is often the result of a missing index on the join column of the inner table.

Pro tip: Paste your EXPLAIN output into explain.depesz.com for a colour-coded, human-readable breakdown. It highlights the most expensive nodes automatically.

Step 4: Audit Your Indexes

Missing indexes are responsible for the majority of PostgreSQL performance problems I encounter. Here's how to find tables with sequential scans that should have indexes:

-- Tables with high sequential scan counts (candidates for indexing)
SELECT
  schemaname,
  tablename,
  seq_scan,
  seq_tup_read,
  idx_scan,
  n_live_tup
FROM pg_stat_user_tables
WHERE seq_scan > 0
  AND n_live_tup > 10000
ORDER BY seq_tup_read DESC
LIMIT 20;

A table that has a high seq_scan count and a low or zero idx_scan count, with many live tuples, is a prime candidate for a new index. Cross-reference with your pg_stat_statements results to find the query causing the scans.

Also check for unused indexes — indexes that consume space and slow down writes without being used for reads:

-- Indexes that have never been used (candidates for removal)
SELECT
  schemaname,
  tablename,
  indexname,
  pg_size_pretty(pg_relation_size(indexrelid)) AS index_size,
  idx_scan
FROM pg_stat_user_indexes
WHERE idx_scan = 0
  AND indexrelname NOT LIKE '%pkey%'
ORDER BY pg_relation_size(indexrelid) DESC;

Step 5: Check for Table Bloat

PostgreSQL's MVCC system means that deleted and updated rows aren't immediately removed — they become "dead tuples" that take up space and slow down queries. If autovacuum isn't keeping up, bloat can significantly degrade performance.

-- Tables with significant dead tuple bloat
SELECT
  schemaname,
  tablename,
  n_live_tup,
  n_dead_tup,
  round(100 * n_dead_tup::numeric / nullif(n_live_tup + n_dead_tup, 0), 1) AS dead_pct,
  last_autovacuum,
  last_autoanalyze
FROM pg_stat_user_tables
WHERE n_dead_tup > 1000
ORDER BY dead_pct DESC
LIMIT 20;

A table with more than 10–20% dead tuples needs attention. For immediate relief, run VACUUM ANALYZE tablename;. For long-term health, you need to tune your autovacuum settings for that table's write pattern.

Tuning autovacuum for high-churn tables

The default autovacuum settings work well for small tables but are too conservative for large, frequently-updated tables. You can set per-table autovacuum parameters:

-- Make autovacuum more aggressive for a high-churn table
ALTER TABLE orders SET (
  autovacuum_vacuum_scale_factor = 0.01,   -- vacuum when 1% of rows are dead
  autovacuum_analyze_scale_factor = 0.005, -- analyze when 0.5% of rows change
  autovacuum_vacuum_cost_delay = 2         -- reduce delay between vacuum chunks
);

Step 6: Check Connection Count and Pooling

PostgreSQL has a hard limit on connections (max_connections, typically 100 by default on managed services). When you approach this limit, new connections queue or fail, and existing connections slow down due to resource contention.

-- Current connection usage by state
SELECT
  state,
  count(*) AS connections
FROM pg_stat_activity
GROUP BY state
ORDER BY connections DESC;

-- Check against max_connections
SHOW max_connections;

If you're regularly using more than 70–80% of max_connections, you need connection pooling. PgBouncer in transaction mode is the standard solution — it multiplexes hundreds of application connections onto a much smaller set of actual PostgreSQL connections.

Putting It All Together

PostgreSQL performance diagnosis is a process of elimination. Start with what's running right now (pg_stat_activity), find your worst offenders over time (pg_stat_statements), understand their execution plans (EXPLAIN ANALYZE), and then work through the underlying causes: missing indexes, bloat, or connection exhaustion.

In my experience, 80% of PostgreSQL performance problems come down to two things: missing or wrong indexes, and autovacuum not keeping up with table bloat. Fix those two and most slowdowns disappear. The remaining 20% are usually query design problems or hardware constraints — and those require a closer look at the specific query patterns.

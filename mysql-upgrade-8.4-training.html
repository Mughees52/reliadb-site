Executive Context 03 / 24

Why Upgrade Now

Support RiskMySQL 8.0 no longer receives standard fixes after EOL.
Security RiskUnpatched CVEs and compliance exposure increase over time.
Operational Risk8.4 introduces breaking behavior in auth, SQL, and configuration.

Treat this as a controlled release program, not a routine maintenance patch.

Compatibility Delta 04 / 24

High-Impact 8.0 to 8.4 Changes

  • Auth: `mysql_native_password` removed; migrate users to `caching_sha2_password`
  • AUTO_INCREMENT: no longer supported on `FLOAT/DOUBLE`; schema changes required
  • SQL semantics: stricter `sql_mode`; implicit `GROUP BY` sorting removed
  • Replication syntax: MASTER/SLAVE commands fully removed; use SOURCE/REPLICA
  • Config hygiene: removed params in `my.cnf` can block startup completely
Program Design 05 / 24

Upgrade Success Formula

  • Prepare: backups + restore proof + checker remediation
  • Validate: replay production workload and full dry-run in staging
  • Execute: prefer replication-based cutover for minimal downtime
  • Protect: pre-approved rollback path with measured recovery time
  • Stabilize: 48-hour monitoring and post-upgrade housekeeping
Pre-Upgrade 06 / 24

Backup Strategy: Two Layers Minimum

  • Logical backup: `mysqldump`, `mydumper`, or `util.dumpInstance`
  • Physical backup: XtraBackup, LVM snapshot, or cold file copy
  • Non-negotiable: restore test on a separate server before production upgrade
  • Evidence: capture backup duration and restore duration for runbook accuracy
  • Goal: guarantee recoverability before any binary/package changes
Pre-Upgrade 07 / 24

Disk Space Planning Rules

2.5x DatadirRecommended free space baseline before major upgrade starts.
+10-20%Temporary working space for dictionary and redo transitions.
Rollback WindowRetain binlogs long enough to recover pre-cutover writes.

Large datasets magnify rollback time; space planning is a reliability control, not capacity trivia.

Pre-Upgrade 08 / 24

Data Integrity and Upgrade Checker

  • Run `mysqlcheck --all-databases --check` and fix corruption before upgrade day
  • Use MySQL Shell `checkForServerUpgrade` against target 8.4.x and your real `my.cnf`
  • Gate: zero unresolved checker errors before Go decision
  • Address warnings for keywords, auth plugins, charset, temporal formats, and constraints
  • Store reports in the change ticket as audit evidence
Compatibility 09 / 24

Configuration and SQL Behavior Changes

  • Remove deprecated parameters (`default_authentication_plugin`, `expire_logs_days`, and others)
  • Explicitly set changed defaults (`innodb_flush_method`, `innodb_io_capacity`, AHI, log buffer)
  • Audit zero-date usage and `ONLY_FULL_GROUP_BY` query behavior in application SQL
  • Rewrite scripts using old replication terminology and command forms
  • Prevent startup failures by testing final `my.cnf` in staging first
Authentication 10 / 24

Auth Migration and ProxySQL Dependencies

User Account Migration

  • Migrate legacy users to `caching_sha2_password`.
  • Validate connector versions for Java, Python, Node, Go, and PHP.
  • Use `mysql_native_password = ON` only as a short transition bridge.

ProxySQL Path

  • Upgrade ProxySQL to 2.5.4+ before MySQL cutover.
  • Enable TLS or RSA public key exchange on backend auth path.
  • Test full chain: App -> ProxySQL -> MySQL 8.4 in staging.
Testing Strategy 11 / 24

Testing Is the Main Risk Reduction Investment

  • Mirror production with two environments: 8.0 baseline and 8.4 target
  • Replay real peak-hour workload, not synthetic-only traffic
  • Test read-only and read-write paths independently
  • Validate ecosystem tools: PMM, Toolkit, XtraBackup, Orchestrator, connectors
  • Use findings to update cutover sequencing and fallback triggers
Testing Strategy 12 / 24

pt-upgrade Workflow

  • Capture 1-3 hours of representative production slow logs during peak periods
  • Build digest and replay side-by-side against 8.0 and 8.4 targets
  • Review query errors, row-count differences, and latency regressions
  • Run both RO and RW modes for broader semantic coverage
  • Classify and remediate all critical deltas before deployment approval
Testing Strategy 13 / 24

Dry-Run and Canary Validation

  • Execute full end-to-end upgrade on staging with exact runbook commands
  • Measure actual timing against planned maintenance window constraints
  • For critical workloads, route 5-10% read traffic to 8.4 canary replica first
  • Observe correctness, latency, and error rates for 24-48 hours
  • Promote only after canary confidence and app-team signoff
Execution Paths 14 / 24

Choose the Upgrade Approach by Risk Profile

Replication-BasedBest for production-critical systems: minutes downtime, fastest rollback.
Rolling ReplicaGreat for existing replica topologies, moderate complexity, low disruption.
In-PlaceLower hardware cost, but highest downtime and rollback burden.

Default recommendation: replication-based approach for large or revenue-critical environments.

Execution Paths 15 / 24

Approach 1: Replication-Based Upgrade

  • Build fresh 8.4 server, restore from 8.0 backup, and start replication
  • Validate app behavior against replica while source remains authoritative
  • At cutover: set source read-only, wait for zero lag, stop replica channel
  • Switch application endpoint to 8.4 and run smoke tests immediately
  • Rollback: redirect traffic back to untouched 8.0 source if needed
Execution Paths 16 / 24

Approach 2: In-Place Upgrade (High Caution)

  • Enforce clean shutdown with `innodb_fast_shutdown = 0` before package replacement
  • On Debian/Ubuntu use `apt-get remove`, never `purge`, to protect datadir
  • Install 8.4 binaries, update `my.cnf`, and start using existing data directory
  • Review error logs for dictionary/redo issues and plugin incompatibilities
  • Rollback: restore backup (potentially 10-20+ hours on large datasets)
Execution Paths 17 / 24

Approach 3: Rolling Replica Upgrade

  • Upgrade replicas one by one; monitor each for stability 24-48 hours
  • Keep original source unmodified as rollback anchor during transition
  • After confidence, promote upgraded replica to source
  • Upgrade former source last to complete topology convergence
  • Ideal when extra hardware is limited but replication topology already exists
Recovery Planning 18 / 24

Rollback Playbook

Preferred Path

  • Replication-based cutovers can revert by traffic redirection.
  • Original 8.0 source remains canonical until confidence achieved.
  • Near-zero data risk when rollback is immediate.

Fallback Path

  • In-place failure requires full restore from pre-upgrade backup.
  • 8.4-generated writes cannot be replayed back to 8.0 safely.
  • Rollback duration scales with data size; plan worst-case windows.
Validation 19 / 24

First 30 Minutes After Cutover

  • Confirm version, startup health, and authentication across all service users
  • Inspect error log warnings and replication status (if applicable)
  • Run smoke tests for critical application transactions
  • Refresh optimizer statistics on key tables to reduce plan drift
  • Enable intensified monitoring on latency p95/p99 and error budgets
Operations 20 / 24

Change Management Essentials

  • Define clear Go/No-Go criteria with owners before maintenance starts
  • Collect explicit sign-off from DBA, app lead, SRE, and change manager
  • Send milestone communication: pre-window, start, cutover complete, all-clear
  • Track actual vs planned durations to improve future upgrade programs
  • Close with PIR: lessons learned, issues, and permanent runbook updates

Troubleshooting 22 / 24

Common Failure Patterns to Avoid

  • Startup blocked by removed parameters left in `my.cnf`
  • Redo-log incompatibility from unclean shutdown pre-upgrade
  • Datadir loss from package purge on Debian/Ubuntu
  • Post-cutover auth failures due to unmigrated legacy users
  • Query regressions from SQL mode and optimizer behavior shifts
Execution Control 23 / 24

Upgrade Checklist (Fast Read)

  • Backups done + restore tested + space validated
  • Upgrade checker errors resolved; warnings reviewed
  • Auth migration and deprecated syntax remediation complete
  • Staging dry-run and app-team signoff complete
  • Rollback rehearsal timed and documented
Closeout 24 / 24

Key Takeaway

The safest path to MySQL 8.4 is a disciplined, test-heavy migration with a pre-proven rollback. Reliability is designed before the maintenance window begins.
